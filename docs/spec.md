# ТЗ: Telegram AI-ассистент для извлечения задач и договорённостей

> Репозиторий: `donskikhmaksim/tg-ai-assistant`. Документ самодостаточный — можно передавать целиком в другой чат/сессию как задание на реализацию.

## 1. Цель

Бот в Telegram, который читает переписку пользователя — личку (входящие **и собственные исходящие**) и группы — батчами прогоняет её через локальную LLM (триаж) и Claude (разбор), извлекает задачи / договорённости / обещания и создаёт их в TickTick в нужном проекте. Обработка **не realtime**, пакетная. Идея связки: дешёвый локальный триаж отсеивает мусор, чтобы не жечь токены Claude.

## 2. Ключевые архитектурные факты (проверено — учитывать жёстко)

- **Личка** мониторится через **Telegram Business**: один бот, подключённый к личному Premium-аккаунту. Business-бот получает событие `business_message` для **обоих** участников — собеседника и **самого владельца** (исходящие). Своё отличается сравнением `from.id == business_connection.user.id`.
- **Business работает только в приватных 1-на-1 чатах.** Группы он не отдаёт.
- **Группы** мониторятся **тем же ботом**, добавленным в группу как участник, с **выключенным privacy mode** (или админ) → обычные `message` апдейты.
- **Истории нет.** Ни в личке, ни в группах бот не видит сообщений до подключения/вступления, и даже своё «послеподключенческое» прошлое перечитать из Telegram нельзя — апдейт приходит один раз. **Единственный архив — наша БД.** «Реран» работает только по сохранённым записям, не повторным запросом к Telegram.
- **Партиционирование по `chatId`.** Личка = id собеседника, группа = id группы. Каждый чат — отдельный контекст; «каши» от параллельных сообщений нет, т.к. у каждого сообщения свой `chatId`.
- **Claude НЕ ходит в MCP напрямую.** Claude возвращает только структурный JSON. Создание задач в TickTick делает **бэкенд как MCP-клиент** к уже существующему Railway-серверу `ticktick-mcp`, **переиспользуя заведённые там токены** (новый OAuth к TickTick не нужен).
- **Premium на аккаунте есть. TTL сырья = 30 дней.**
- **Уведомлений в Telegram нет.** Результат = созданная задача в TickTick, и всё. Бот нужен только как пульт для привязки проектов.

## 3. Стек

- **Python 3.12**
- **aiogram 3.x** (полноценная поддержка `business_connection` / `business_message`)
- **MongoDB** + **Motor** (async)
- **APScheduler** — батч-обработчик по расписанию
- **Anthropic SDK** (`anthropic`) — Claude, модель `claude-opus-4-8`, структурный вывод
- **OpenAI SDK** (`openai`) с `base_url` на локальный **Ollama** — Qwen-триаж
- **MCP-клиент** к `ticktick-mcp` (питоновский `mcp` пакет, HTTP/SSE-транспорт — см. §12)
- Хостинг бэка: Railway (рядом с MCP). Qwen — локально на Mac mini.

## 4. Архитектура (поток)

```
Telegram
  ├─ business_message  (личка: вход + исходящие)   ─┐
  └─ message           (группы, privacy off)        ─┤→ один бот / один бэк
                                                      ↓ сохраняем КАЖДЫЙ апдейт сразу
                                          Mongo: raw_messages (TTL 7д, ключ chatId)
                                                      ↓ APScheduler, каждые 30 мин
                       для каждого «грязного» чата → текущее ОКНО РАЗГОВОРА (см. §7)
                                                      ↓
                       Tier 1 — Qwen (Ollama, локально): в окне есть задача? да/нет
                                                      ↓ только «да»
                       Tier 2 — Claude (opus-4-8): окно + уже известные задачи чата
                                                  → инкрементальный JSON (новые/изменённые)
                                                      ↓
                       резолв проекта: chat_project_map[chatId] ?? Inbox
                                                      ↓ дедуп
                       TickTick через Railway-MCP (create_task / update), старые токены
```

## 5. Модель данных (MongoDB)

**`raw_messages`** — сырьё, **TTL-индекс на 30 дней** (`expireAfterSeconds` по `date`):
```jsonc
{
  "chatId": "user_12345" | "group_-100123",
  "type": "dm" | "group",
  "direction": "in" | "out",          // out = исходящее владельца
  "fromId": "67890",
  "senderName": "…",
  "text": "…",
  "messageId": 555,
  "businessConnectionId": "…|null",
  "date": ISODate
}
```
Индексы: `{chatId:1, date:1}`, TTL на `date`.

**`tasks`** — постоянное (НЕ затирать):
```jsonc
{
  "chatId": "user_12345",
  "task": "Прислать договор",
  "who": "me" | "counterparty",
  "counterpartyName": "…|null",
  "deadline": "2026-06-28|null",
  "status": "open|done|cancelled",
  "sourceMessageIds": [555, 556],
  "dedupHash": "sha1(chatId + normalized_task)",
  "ticktickTaskId": "…|null",
  "projectId": "…",
  "createdAt": ISODate,
  "updatedAt": ISODate
}
```
Индексы: `{chatId:1}`, уникальный `{dedupHash:1}`.

**`chat_project_map`** — привязка чат → проект:
```jsonc
{ "chatId": "user_12345", "ticktickProjectId": "abc", "projectName": "Клиент X" }
```

**`chat_state`** — служебное состояние обработки на чат:
```jsonc
{ "chatId": "user_12345", "lastMessageAt": ISODate, "lastProcessedAt": ISODate }
```

**`chat_summary`** — ДОЛГОВРЕМЕННАЯ ПАМЯТЬ по чату (НЕ затирать, TTL не применяется):
```jsonc
{
  "chatId": "user_12345",
  "summary": "Сжатая выжимка: о чём шла речь, договорённости, нерешённые вопросы, кто кому что должен, важные факты/предпочтения собеседника.",
  "updatedAt": ISODate
}
```
Назначение: сохранять смысл переписки независимо от TTL сырья. Маленькое, живёт вечно. Решает кейс «тему обсуждали неделю назад, сырьё уже затёрто, а сейчас к теме вернулись» — предыстория берётся отсюда.

**`bot_state` / `settings`** — состояние диалогов бота, business_connection (id владельца) и т.п.

## 6. Telegram-интеграция

**@BotFather:**
- `/newbot` → `BOT_TOKEN`
- Bot Settings → **Business Mode → Enable**
- Bot Settings → **Group Privacy → Turn off**

**Подключение лички:** Settings → Telegram Business → Chatbots → юзернейм бота → выдать права (чтение/ответ/управление) → область чатов «все».

**Хендлеры:**
- `business_connection` → сохранить `business_connection.id` и `user.id` (это «я», для определения direction).
- `business_message`, `edited_business_message`, `deleted_business_messages` → личка → `raw_messages` (`type:"dm"`, `direction` по `from.id == connection.user.id`).
- `message` в группах → `raw_messages` (`type:"group"`, `direction:"out"` если `from.id == owner_id`, иначе `in`).
- **Сохраняем апдейт атомарно до любой обработки.** Потеря безвозвратна — повторно из Telegram не придёт. На каждое сохранение обновляем `chat_state.lastMessageAt`.

## 7. Пайплайн обработки (батч) — ОКНА РАЗГОВОРА

Главная задача: переписку, растянутую на часы, не резать посередине и не терять контекст — ни на стыке прогонов, ни через недели.

**Важно — два разных механизма, не путать:**
- **Окно разговора (gap/lookback ниже)** отвечает ТОЛЬКО за «какие свежие сырые сообщения смотреть сейчас». Оно НЕ отвечает за «помнить историю».
- **Долговременная память** (`chat_summary` + открытые задачи из `tasks`) отвечает за предысторию и НЕ зависит от TTL/окна. На каждый разбор Claude получает окно **плюс** эту память. Поэтому тема, всплывшая через неделю (когда сырьё уже затёрто по TTL), всё равно разбирается с полным контекстом — он берётся из резюме.

**Параметры (дефолты):**
- `BATCH_INTERVAL_MIN = 30` — частота прогона
- `CONV_GAP_HOURS = 6` — пауза, после которой разговор считается «закрытым»
- `MAX_LOOKBACK_HOURS = 48` — жёсткий потолок размера окна
- `RAW_TTL_DAYS = 30` (>> MAX_LOOKBACK, чтобы окно всегда было целым + большой запас на реран)

**Алгоритм каждые 30 минут:**
1. Берём список **«грязных» чатов** — где `chat_state.lastMessageAt > chat_state.lastProcessedAt` (появились новые сообщения с прошлого прогона). Чаты без новых сообщений пропускаем.
2. Для каждого грязного чата строим **текущее окно разговора**: берём сообщения этого `chatId`, идём назад от последнего, пока не встретим паузу > `CONV_GAP_HOURS` или не упрёмся в `MAX_LOOKBACK_HOURS`. Это всё «живое» окно целиком.
   - Кейс «я → через час он → через час я» (6 кусков по ~1 ч): между кусками нет паузы > 6 ч → всё попадает в **одно окно**. Стыков «не хватило куска» не бывает — контекст всегда полный.
3. **Tier 1 — Qwen (Ollama):** на всё окно → строгий JSON `{ "has_task": bool }`. Если `false` — обновляем `lastProcessedAt` и идём дальше.
4. **Tier 2 — Claude `claude-opus-4-8`** (только если `has_task=true`):
   - Вход: (а) полный текст окна с разметкой кто `in`/`out` и временем; (б) **долговременная память чата** — `chat_summary.summary` + список открытых задач этого чата из `tasks`.
   - Просим **инкрементально** и вернуть три вещи: новые задачи, изменения статусов существующих (обещание выполнено/отменено по более поздним сообщениям) и **обновлённое резюме чата**.
   - Структурный вывод (`output_config.format`, json_schema):
     ```jsonc
     {
       "new_tasks": [ { "task": "...", "who": "me|counterparty", "counterpartyName": "...|null",
                        "deadline": "YYYY-MM-DD|null", "suggested_project": "...|null",
                        "source_message_ids": [555,556] } ],
       "status_updates": [ { "task_match": "текст/идентификатор", "new_status": "done|cancelled" } ],
       "updated_summary": "обновлённая сжатая выжимка чата с учётом этого окна"
     }
     ```
   - Включить **prompt caching** на стабильный системный промпт.
   - После ответа: записать `updated_summary` в `chat_summary` (вот так смысл переезжает в память до того, как сырьё затрётся по TTL).
5. **Дедуп:** для каждой new_task считаем `dedupHash = sha1(chatId + normalized(task))`. Если такой уже есть — пропускаем (уникальный индекс страхует). Так повторный анализ перекрывающихся окон не плодит дубли.
6. **Резолв проекта:** `chat_project_map[chatId]` → иначе **Inbox**. `suggested_project` от Claude — лишь подсказка, финально решает бэк.
7. **TickTick:** бэк как MCP-клиент → создать задачу (`create_task`), сохранить `ticktickTaskId`. По `status_updates` — обновить/закрыть соответствующую задачу в TickTick.
8. Обновить `chat_state.lastProcessedAt = now`.

**Реран (ручной):** команда/кнопка пересобирает окно по `chatId` из БД и гоняет заново. Работает, пока сырьё живо в пределах TTL.

**Стоимость:** держится низкой за счёт (а) обработки только грязных чатов, (б) Qwen-ворот перед Claude, (в) инкрементального промпта (Claude видит уже извлечённое и не дублирует рассуждения), (г) prompt caching.

## 8. Qwen (локально)

- **Mac mini M2 Pro 32 GB**, **Ollama**.
- Модель **Qwen2.5-32B-Instruct** (≈ та самая «34-миллиардная»). На 32 GB в 4-bit ≈ 18–20 GB, влезает; скорость ~8–12 ток/с — для батча нормально.
  - Фолбэк, если будет тесно/медленно: Qwen2.5-14B-Instruct.

## 9. Бот: UX управления (привязка проектов)

Уведомлений о созданных задачах НЕТ (задачи просто появляются в TickTick). Бот нужен для привязки чат → проект.

**Фаза 1 (делаем сразу):**
- `/start` → приветствие + **постоянное нижнее reply-меню**: `🔗 Привязать проект`, `📋 Мои привязки`, `❌ Отвязать`.
- **Привязка:** бот тянет проекты через MCP (`get_projects`/`list_projects`) → показывает **инлайн-кнопками** → выбор → запись в `chat_project_map`. В группе — команда `/bind`.

**Фаза 2 (позже):**
- Кнопка `⚙️ Настройки` открывает **Telegram WebApp** (мини-аппа): HTML с **дропдаунами** (чат/группа ↔ проект TickTick), «Сохранить» → данные обратно боту → запись маппинга. Списки проектов с бэка (он из MCP).

## 11. Нефункциональные / нюансы

- Сохранять апдейты атомарно до любой обработки; терять нельзя.
- TTL не агрессивный (30 дней, >> MAX_LOOKBACK) — чтобы окно всегда было целым и был большой запас на реран. Контекст за пределами TTL не теряется — он живёт в `chat_summary`.
- Идемпотентность: повторный прогон не плодит дубли (дедуп-хеш + инкрементальный промпт).
- Мультиязычность: переписка RU/EN — промпты Qwen и Claude должны это учитывать.
- Группы: та же логика извлечения, что и в личке.
- Приватность: чужие сообщения в группах гоняются через облачный Claude — для личного использования ок, но держать в голове.

## Примечания по реализации (resolved during implementation)

- **§12.1 разрешено:** `ticktick-mcp` — транспорт **Streamable HTTP** на `/mcp/<secret>`;
  тулзы подтверждены по коду сервера: `get_projects`, `create_task(title, project_id, …)`,
  `update_task(task_id, project_id, …)`, `complete_task(project_id, task_id)`. Тулзы
  возвращают **человекочитаемые строки, не JSON** — бэкенд парсит `ID:`/`Name:` строки.
- **TTL сырья = 30 дней** (в схеме §4 в диаграмме осталось «7д» — это опечатка; корректно 30,
  как в §2/§5/§7/§11).
- **BotFather:** в актуальном UI «Business Mode» называется **Secretary Mode**.
- **Claude:** `output_config.format` (json_schema) — не устаревший `output_format`;
  adaptive thinking + prompt caching на стабильном системном промпте.
